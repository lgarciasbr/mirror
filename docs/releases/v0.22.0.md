# v0.22.0 — Mode Orientation and Test Hygiene

**Date:** 2026-06-06

## Highlights

- Mirror now has explicit mode lifecycle for `◌ Mirror Mode`, `■ Builder Mode`, and `△ Explorer Mode`.
- The Pi status line shows the current mode and active journey when present.
- Mode changes render compact transition surfaces in the conversation.
- New sessions start in `◌ Mirror Mode` from present intention, without inheriting stale Builder/Explorer mode or the last journey.
- Explorer Mode exists as an experimental journey lens while the full exploratory behavior is being built.
- `/mm-discard` deletes the current runtime conversation before quitting a test session.

## Where We Started

Explorer Mode needed a foundation before its exploratory behavior could be built.
As soon as a third lens entered the product language, two older ambiguities became
visible: Mirror did not clearly show which lens was active, and runtime test
sessions polluted the personal conversation database.

The first implementation also revealed a subtle product distinction: a new Mirror
session is an act of intention, not an assumption of continuity with the previous
journey. Continuity belongs inside an active session or an explicitly reopened
journey.

## What Changed

`memory mode activate|deactivate|status` now stores the active operating lens.
`memory mirror load`, `memory build load`, and `memory explore load` activate the
corresponding lenses and feed the Pi status line through `welcome --status-line`.

Mode transition surfaces now orient the user at the moment of lens change:
Mirror Mode shows identity, journey, persona routing, and available lenses;
Builder Mode shows journey context, project path, briefing, and the construction
boundary; Explorer Mode shows an experimental availability note and the
uncertainty-preservation boundary.

Pi session start now clears stale explicit mode and sticky journey context, so a
fresh session opens as `◇ alisson-vale · ◌ Mirror Mode · ✓` unless the user
activates a journey again.

The new `/mm-discard` skill calls `conversation-logger discard-current`, deletes
the current conversation through the existing deletion service, and marks the
runtime session so the assistant confirmation is not logged as a replacement
conversation.

## Conscious Exclusions

- This release does not implement the full Explorer Mode story-thickening engine.
- Explorer Mode persistence, Signal Radar, Narrative Field Snapshot, and Builder promotion remain future CV16 work.
- `/mm-discard` does not force-close Pi; it discards the conversation and tells the user to quit.
- Discarding a conversation follows existing deletion semantics: extracted memories are detached, not deleted.

## What We Learned

Mode and journey are different axes. Mode is the lens of operation; journey is
situated intention. They can meet in the status line, but their lifecycles should
not be conflated.

We also learned that transition surfaces and status bars solve different
problems. The status bar preserves orientation over time; the transition surface
makes the moment of change visible.

## Next Horizon

The next CV16 work can move from orientation to behavior: Explorer activation
contract, in-session exploratory state, and reliable Story Thickened rendering.
