# v0.10.5 — Cumulative Release Notes

**Date:** 2026-05-25

## Highlights

- Added `runtime release-notes pending` to list all releases newer than the
  current runtime version.
- Added `--from <version>` for explicit comparison, useful for smoke tests and
  support flows.
- Added `--ref <git-ref>` so the command can read release notes from a specific
  source such as `origin/stable` or `HEAD`.
- Kept the existing `runtime release-notes latest` and explicit version behavior
  unchanged.
- Made Pi assistant-message logging fire-and-forget so Mirror persistence does
  not sit on Pi's `agent_end` lifecycle path.

## Why This Patch Exists

After the Web Visibility release train, production validation showed a gap in the
user experience: when a user is several versions behind, showing only the latest
release note is not enough. They need the whole path between their installed
version and the current stable release.

## What Changed

The runtime release notes command now supports a pending mode:

```bash
uv run python -m memory runtime release-notes pending
```

For diagnostics and validation, the comparison can be made explicit:

```bash
uv run python -m memory runtime release-notes pending --from 0.10.2 --ref HEAD
```

The output groups each pending release with its digest and highlights, preserving
the concise release-note style while making cumulative updates visible.

This patch also keeps the Mirror logger aligned with that lifecycle principle:
assistant response persistence now runs in the background instead of being
awaited inside Pi's `agent_end` extension hook.

## Next Horizon

The pending release flow should become the default companion to runtime update
notices. A future polish pass can wire it directly into welcome/update guidance
so users can ask what changed before applying an update.
